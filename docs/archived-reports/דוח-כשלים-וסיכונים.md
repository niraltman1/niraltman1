## קריטי (תוקן במסגרת ביקורת זו)

### K1 — `require()` ב-ESM module בזמן ריצה
**קובץ:** `packages/database/src/hardening.ts:112`
**תיאור:** `backup()` השתמשה ב-`require('node:fs')` בתוך ESM module. כל קריאה ל-`backup()` הייתה קורסת ב-runtime עם `ReferenceError: require is not defined`.
**השפעה:** כשל בכל תהליכי ה-backup. נתונים ללא גיבוי.
**תיקון:** הועבר ל-static import בראש הקובץ. ✅

### K2 — עמוד השחזור ריק תמיד (עמודות שגויות ב-SQL)
**קובץ:** `packages/api/src/routes/recovery.ts`
**תיאור:** Route GET /agents ביצע `SELECT locked_at` ו-POST /clear-locks כתב ל-`completed_at`. עמודות אלו לא קיימות ב-schema (migration 049 — השדות הם `started_at` ו-`finished_at`). SQLite זרק שגיאה, ה-catch החזיר רשימה ריקה תמיד.
**השפעה:** לוח השחזור הציג "אין נעילות" גם כשסוכנים רצים. clear-locks תמיד החזיר `clearedCount: 0`.
**תיקון:** תוקן לשמות העמודות הנכונים. ✅

### K3 — `/api/tabular/ingest` החזיר 404 תמיד
**קובץ:** `packages/api/src/app.ts`
**תיאור:** `tabular.ts` route קיים ומלא, אך לא נרשם ב-`createApp()`. כל קריאה ל-`POST /api/tabular/*` קיבלה 404.
**השפעה:** קליטת CSV/Excel מושבתת לחלוטין. הקוד כולו חסר שימוש.
**תיקון:** נוסף import + `app.use('/api/tabular', tabularRouter(repos))`. ✅

### K4 — crash handlers רשומים אחרי הפעלת הAPI
**קובץ:** `packages/api/src/start.ts`
**תיאור:** `process.on('uncaughtException', ...)` נרשם בסוף הקובץ, אחרי `new MigrationRunner().run()`, אחרי כל ה-repos, ואחרי `app.listen()`. קריסה בזמן מיגרציה (שגוי מאוד) — לא תיקלט ב-SystemEvents.
**השפעה:** crash events לא נשמרים בשלב הקריטי ביותר של ההפעלה.
**תיקון:** הוקמו handlers לפני כל startup. `_earlyDb` מאפשר כתיבה ל-SystemEvents ברגע שה-DB זמין. ✅

### K5 — נעילות סוכנים עצומות מ-crash קודם
**קובץ:** `packages/api/src/start.ts`
**תיאור:** לא היה cleanup של `AgentRunRegistry` בעת הפעלה מחדש. כל agent שהיה `status='running'` בעת קריסה נשאר חסום לצמיתות (לא ניתן להפעיל שוב לאותו תיק).
**השפעה:** לאחר קריסה, agent לא ניתן להפעלה ללא ניקוי ידני דרך `/admin/recovery`.
**תיקון:** נוסף UPDATE אוטומטי בעת startup. ✅

## גבוה (לא תוקן — דורש מחזור פיתוח)

### G1 — `execFileSync` חוסם Event Loop ב-OCR
**קובץ:** `packages/pipeline/src/ocr-service.ts`
**תיאור:** כל 4 שלבי ה-OCR משתמשים ב-`execFileSync` — חוסם את ה-event loop של Node.js למשך דקות עבור PDF כבד.
**השפעה:** API לא מגיב לשאר הבקשות בזמן OCR. במקרה של 3 קבצים במקביל (BATCH_SIZE=3) — עצירה מוחלטת.
**מומלץ:** מעבר ל-`execFile` + worker_threads או subprocess.

### G2 — `WorkflowIdempotencyLog` ללא TTL
**קובץ:** `packages/orchestrator/src/orchestrator.ts`
**תיאור:** `acquireLock()` מוסיף שורה ל-`WorkflowIdempotencyLog`. אם הקורא קורס לפני `releaseLock()`, המנעול נשאר לצמיתות. אין מנגנון expiry.
**השפעה:** מסמך יכול להיחסם לצמיתות.
**מומלץ:** הוסף `acquired_at` ובקרת TTL (e.g. 2h).

### G3 — חיבור DB יחיד לכל ה-workers
**קובץ:** `packages/api/src/start.ts`
**תיאור:** כל ה-repos ו-6 ה-workers משתמשים בחיבור DB יחיד (`DatabaseConnection`). Better-sqlite3 סינכרוני — אין race condition, אך עמוד WAL אחד + ריצות OCR ארוכות עלולות ליצור contention.
**השפעה:** ביצועים נמוכים בעומס גבוה. לא בעיה אצל עו"ד יחיד.
**מומלץ:** שמור כך לבטא — בחן מחדש ב-v2.

### G4 — אין validation קלט ב-Zod על כל routes
**תיאור:** חלק מה-routes (למשל `/api/recovery/event`) לא השתמשו ב-Zod לvalidation מלא. תוקן חלקית בביקורת זו.
**מומלץ:** סרוק כל routes + הוסף Zod schema.

### G5 — rate limiting ל-AI routes לא מספיק
**תיאור:** `/api/agents/*` כפוף ל-300 req/min global. כל agent יכול לקחת 45 שניות. 7 בקשות מקביליות = event loop תפוס.
**מומלץ:** הוסף rate limiter ייעודי: max 3 req/min ל-`/api/agents/`.

### G6 — CORS מרשה `localhost:5174`
**קובץ:** `packages/api/src/app.ts`
**תיאור:** `origin: ['http://localhost:5174', ...]` — port 5174 הוא port backup של Vite. ב-production זה לא אמור להיות פתוח.
**מומלץ:** ב-production הסר `localhost:5174` (רק `localhost:3001` אמור לשרת dashboard).

## בינוני

### B1 — ה-dashboard לא מוגן מאחורי authentication
**תיאור:** אין middleware שמחייב login לפני גישה ל-React SPA. ה-API עצמו מאובטח (requireAuth), אך תיאורטית ניתן לפתוח את הדשבורד ללא auth.
**השפעה:** נמוכה — הAPI מוגן. בפרסום לרשת פנימית יש לוודא.

### B2 — `seedDefaultAdmin` מוסיף admin בכל הפעלה
**קובץ:** `packages/api/src/middleware/auth.ts`
**תיאור:** `seedDefaultAdmin()` נקרא בכל הפעלת API. אם password ברירת מחדל פשוט — סיכון אם DB זמין ברשת.
**מומלץ:** וודא שה-seed בודק אם admin קיים + דרוש שינוי password בהפעלה ראשונה.

### B3 — אין timeout על MigrationRunner
**תיאור:** מיגרציה שתוקעת (e.g. VACUUM על DB גדול) תחסום את הפעלת ה-API ללא הודעה.
**מומלץ:** הוסף watchdog timeout.

### B4 — Backup לא מאמת שלמות אחרי copy
**קובץ:** `packages/database/src/hardening.ts`
**תיאור:** `backup()` מבצע `copyFileSync` אך לא מאמת checksum לאחר מכן.
**מומלץ:** הוסף `PRAGMA integrity_check` על קובץ ה-backup.

### B5 — `SupportBundleExporter` לא מנקה tmpDir בשגיאה
**תיאור:** אם `zipSingleFile` נכשל, ה-tmpDir לא נמחק.
**מומלץ:** try/finally + cleanup.
