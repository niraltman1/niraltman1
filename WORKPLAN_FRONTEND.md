# תוכנית עבודה — Frontend / UX

> **מטרת-על:** להפוך את הממשק ממערכת "שנבנתה למתכנת" לכלי עבודה שנבנה **לעורך דין**.
> כל מסך נמדד בשאלה אחת: *"האם עו"ד עסוק, תחת לחץ מועדים, מבין בשנייה מה לעשות כאן —
> בלי לדעת מה זה Queue, OCR, embedding או circuit breaker?"*
>
> תוכנית זו משלימה את `WORKPLAN_BACKEND.md` ו-`WORKPLAN_COMMUNICATIONS.md` (מודול תקשורת אומני-ערוצי —
> טלגרם/וואטסאפ/מייל/שיחות, נגיש מהסיידבר + כל כרטיס לקוח + כל תיק). סדר העבודה הכללי: ה-Backend מאכלס נתונים
> ומשלים API → ה-Frontend הופך אותם לחוויית עבודה משפטית. תלויות מסומנות `⟵ B#`.

---

## עקרונות מנחים (חלים על כל שלב)

1. **שפה משפטית, לא הנדסית.** למחוק מהממשק כל מונח טכני. ראה "מילון תרגום" למטה.
2. **מבוסס-מטלה, לא מבוסס-טבלה.** המסך הראשי עונה על "מה דחוף לי היום", לא "הנה כל הרשומות".
3. **מועד הוא המלך.** עו"ד חי לפי תאריכים דיוניים. מועדים, התראות וספירה-לאחור בכל מקום רלוונטי.
4. **מקור תמיד גלוי.** כל תובנת AI / ציטוט פסיקה / מועד — עם "מקור" לחיץ. אפס טקסט "מרחף" בלי מקור.
5. **עברית-first, RTL מלא, רושמה משפטית פורמלית.** מספרי תיק/ת.ז. בפורמט ישראלי תקני.
6. **כשל בחן וברור.** "השירות החכם אינו זמין כעת — המשך ידני" במקום שגיאת רשת/stack trace.
7. **מצבי-ריק מנחים.** מסך ריק מסביר מה לעשות ("עדיין אין מסמכים בתיק — גרור לכאן קובץ").
8. **דיסקרטיות וחיסיון.** אין הצגת תוכן מסמך/שם לקוח בלוגים/טוסטים/כותרות חיצוניות.

### מילון תרגום מונחים (Engineering → עו"ד)
| במערכת היום | יוצג לעו"ד |
|---|---|
| Queue / תור עיבוד | "מסמכים בעיבוד" / "בקליטה" |
| Poisoned items / requeue | "קבצים שנכשלו — נסה שוב" |
| OCR_COMPLETE / CLASSIFIED / ENRICHED | סטטוס ידידותי: "נסרק", "מסווג", "נותח" |
| Embedding / vector / semantic | "חיפוש חכם" (בלי לחשוף את המנגנון) |
| Circuit breaker / Ollama down | "העוזר החכם אינו זמין כרגע" |
| Confidence score 0.78 | "רמת ודאות: גבוהה/בינונית/נמוכה" + צבע |
| Agent / run / token stream | "ניתוח AI" / "מכין סיכום…" |

## Phase F0 — תשתית UX ושפה (שבוע 1)
**למה ראשון:** בלי בסיס שפה ועיצוב אחיד, כל מסך חדש יחזור על אותן טעויות.

- [ ] **מילון מונחים מרכזי** (`apps/dashboard/src/lib/legal-terms.ts`) — מיפוי סטטוסים/מונחים טכניים
      לתוויות עבריות משפטיות. כל המסכים צורכים ממנו (single source of truth).
- [ ] **עיצוב מערכת (design tokens)** — ביקורת על פלטת צבעים, טיפוגרפיה עברית (קריאוּת לטקסט משפטי ארוך),
      ריווח, רכיבי בסיס (כפתור/כרטיס/טבלה/באנר-אזהרה/צ'יפ-מועד) אחידים ונגישים.
- [ ] **רכיבים משותפים חדשים:** `<DeadlineChip>` (ספירה-לאחור צבעונית), `<SourceLink>` (מקור לחיץ),
      `<ConfidenceBadge>` (גבוה/בינוני/נמוך), `<EmptyState>`, `<AsyncBoundary>` (טעינה/שגיאה/ריק אחיד),
      `<HebrewDate>` / `<CaseNumber>` (פורמט ישראלי).
- [ ] **ארכיטקטורת מידע (IA) וניווט מחדש** — לקבץ את הסיידבר סביב מודל-חשיבה של עו"ד:
      *היום שלי · תיקים · לקוחות · מסמכים · מועדים ולוח-שנה · מאגר חקיקה ופסיקה · התראות · הגדרות*.
      להוריד מהניווט הראשי פריטים טכניים/פנימיים (Queue, Vacuum, Mission-Control) → לאזור "מערכת".
- **קבלה:** מסמך עיצוב מאושר + ספריית רכיבים ב-Storybook/דמו + מילון מונחים בשימוש ב-≥3 מסכים.

## Phase F1 — סגירת ה-Stubs (שבועות 2–3)
חמשת המסכים שכרגע ריקים/לא מציגים נתונים. כולם backend-ready (חוץ מהמצוין).

- [ ] **חיפוש גלובלי (`/search`)** — *הפער המביך:* ה-API (FTS5) מחזיר תוצאות אך ה-UI לא מרנדר.
      לבנות תצוגת תוצאות מקובצת (תיקים/מסמכים/לקוחות/פסיקה), הדגשת המונח, ניווט-מקלדת, "אין תוצאות".
- [ ] **לוח שנה (`/calendar`)** — תצוגת חודש/שבוע של דיונים+מועדים, לחיצה → תיק. ⟵ B (calendar feed)
- [ ] **ראדאר מועדים (`/deadlines`)** — רשימה מתועדפת לפי קרבה, צבע לפי דחיפות, סינון לפי תיק/סוג-הליך,
      פעולה מהירה "צור משימה". מוזן מ-Rules_Engine + תיקים. ⟵ B (deadline computation)
- [ ] **אוספים חכמים (`/collections`)** — שמירת חיפוש/פילטר כ"אוסף" + תצוגתו. ⟵ B (collections engine)
- [ ] **ספריית טפסים/Stens (`/stens`)** — דפדוף בתבניות, מילוי מודרך, ייצוא. ⟵ B (templates data)
- **קבלה:** אפס מסך-stub בניווט הראשי; כל מסך עם נתונים אמיתיים או EmptyState מנחה.

## Phase F2 — מסך מאגר חקיקה ופסיקה (שבועות 3–4) ⟵ B0/B1
**הפער שהמשתמש הצביע עליו:** יש API לקורפוס אך אפס UI. בלי זה הנתונים שאוכלסו "בלתי נראים".

- [ ] **מסך "מאגר משפטי"** — שני טאבים: *חקיקה* | *פסיקה*.
- [ ] חיפוש FTS5 + סינון: לפי חוק/ערכאה (עליון/מחוזי/שלום/עבודה/משפחה/תעבורה)/שנה/תחום.
- [ ] **תצוגת טקסט מילולי** — קריאה נוחה לטקסט ארוך בעברית, ניווט-סעיפים, הדגשת מונח-חיפוש.
- [ ] **כרטיס מקור** — מטא-דאטה (מספר תיק, שופטים, צדדים, תאריך, רישיון/מקור-נתונים, דגל
      "רישיון לא מאומת" כשרלוונטי), כפתור "צטט" שמייצר ציטוט ישראלי תקני.
- [ ] **שילוב בתוך התיק** — מתוך פסק/חוק: "קשר לתיק" / "הוסף לתקדימים". חיבור ל-`/precedents` הקיים.
- **קבלה:** עו"ד מוצא פסק דין/סעיף חוק רלוונטי, קורא אותו מילולית, ומצטט אותו לתיק — בלי לצאת מהמערכת.

## Phase F3 — השלמת ה-Partials סביב מחזור-חיי התיק (שבועות 4–6)
- [ ] **פרטי תיק (`/cases/:id`)** — מסך-על אמיתי: סטטוס, צדדים, מועדים קרובים, מסמכים אחרונים,
      משימות פתוחות, סיכון, תקדימים מקושרים. נקודת-הכניסה היומית של עו"ד לתיק.
- [ ] **שולחן עבודה (`workbench`)** — להשלים תוכן 3-הפאנלים (ציר-זמן | צופה-מסמך | תובנות) לשימוש אמיתי.
- [ ] **הכנה לדיון (`hearing-prep`)** — צ'קליסט דיוני מ-Rules_Engine, מסמכים נדרשים, פערי-ראיות,
      "תיק דיון" מודפס/מיוצא.
- [ ] **פרטי לקוח** — ציר-זמן לקוח, תיקים מקושרים, אנשי-קשר, מצב כספי (ledger).
- [ ] **Admin** (journal/recovery/RBAC) — להעביר ל"אזור מערכת" עם שפה ברורה; להשלים פעולות.
- **קבלה:** עו"ד מנהל תיק מקצה-לקצה (פתיחה → מסמכים → מועדים → דיון → סגירה) ממסכי-התיק בלבד.

## Phase F4 — "היום שלי" + חוויית עו"ד (שבועות 6–7)
- [ ] **דף בית מבוסס-מטלה ("היום שלי")** מחליף את dashboard ה-KPI הטכני: מועדים דחופים היום/השבוע,
      משימות, דיונים, מסמכים שנקלטו וממתינים לאישור, התראות. הכל לחיץ ומוביל לפעולה.
- [ ] **תובנות AI בשפת עו"ד** — להחליף טקסט-דמו מקודד (CasesPage) בפלט אמיתי; כל תובנה עם "מקור" + רמת-ודאות
      + באנר "דרושה בדיקת עו"ד" היכן שמתחייב (מועדים/סיכון).
- [ ] **סקירת מסמך עם ציטוטים** — בצופה המסמך: הדגשות מקושרות לחקיקה/פסיקה מהמאגר.
- [ ] **זרימת אישור/דחייה אחידה** לכל פלט-AI (action-plan, תובנות, תבניות) — "אשר / דחה / ערוך + מקור".
- **קבלה:** משתמש-בדיקה (עו"ד) מבצע יום-עבודה מדומה ומדרג מובנוּת/יעילות ≥ סף מוסכם.

## Phase F5 — ליטוש, נגישות, הדפסה וביצועים (שבוע 8)
- [ ] **הדפסה/ייצוא משפטי** — תצוגת-הדפסה נקייה לכתבי-טענות, "תיק דיון", סיכומי-תיק (כבר יש docx — לחבר ל-UI).
- [ ] **נגישות (a11y)** — ניווט-מקלדת מלא, ARIA, ניגודיות, גודל-טקסט; קריטי לטקסט משפטי ארוך.
- [ ] **מצבי שגיאה/טעינה/ריק** אחידים בכל מסך (`<AsyncBoundary>`), בעברית, בלי jargon.
- [ ] **Onboarding ומצב-ריק ראשוני** — מערכת חדשה/ריקה מנחה את עו"ד מה לעשות צעד-אחר-צעד.
- [ ] **ביצועים** — virtualization לטבלאות גדולות, lazy-load למסכים כבדים, מדידת bundle.
- **קבלה:** lint+typecheck נקיים, ביקורת a11y עוברת, אין רגרסיית ביצועים.

## מדדי הצלחה (Definition of Done לתוכנית)
- אפס מסך-stub בניווט הראשי; אפס מונח הנדסי גלוי למשתמש.
- עו"ד מבצע מחזור-תיק מלא + מוצא ומצטט מקור משפטי — בלי תמיכה טכנית.
- RTL/עברית, a11y, הדפסה — עוברים ביקורת.
- כל פלט-AI עם מקור + רמת-ודאות + נתיב אישור/דחייה.
