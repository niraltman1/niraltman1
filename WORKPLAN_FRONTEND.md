# הפאזות שנותרו (העבודה האמיתית)

## Phase F-A — חיווט תוצאות החיפוש הגלובלי *(quick win, שבוע 1)*
**הפער המביך:** ה-API (FTS5) מחזיר תוצאות, ה-hook `useSearch` קיים — אך `SearchPage` לא צורך אותו ולא מרנדר.
- [ ] לחבר `SearchPage.tsx` ל-`useSearch()`; debounce על הקלט.
- [ ] תצוגת-תוצאות מקובצת (תיקים / מסמכים / לקוחות / פסיקה), הדגשת-המונח, ניווט-מקלדת.
- [ ] מצב "אין תוצאות" + מצב טעינה אחיד; קליק → ניווט ליעד.
- **קבלה:** הקלדת מונח מציגה תוצאות אמיתיות מקובצות וניתנות-לניווט.

## Phase F-B — מסך מאגר חקיקה ופסיקה (קריאה מילולית) *(שבועות 1–3)* ⟵ B0/B1
**הפער המרכזי:** הנתונים שאוכלסו (חקיקה #50 / פסיקה #52) **בלתי-נראים** — אין מסך לקריאתם. `PrecedentsPage`
הוא אימות-תקדימים, לא דפדפן-קורפוס.
- [ ] **מסך "מאגר משפטי"** (route חדש, למשל `/library`) — שני טאבים: *חקיקה* | *פסיקה*.
- [ ] חיפוש FTS5 + סינון לפי חוק/ערכאה (עליון/מחוזי/שלום/עבודה/משפחה/תעבורה)/שנה/תחום.
- [ ] **תצוגת טקסט מילולי** — קריאה נוחה לעברית ארוכה, ניווט-סעיפים, הדגשת-מונח.
- [ ] **כרטיס מקור** — מטא (מס' תיק, שופטים, צדדים, תאריך, מקור-נתונים + דגל "רישיון לא מאומת"),
      כפתור "צטט" המייצר ציטוט ישראלי תקני.
- [ ] **שילוב בתיק** — "קשר לתיק" / "הוסף לתקדימים" (חיבור ל-`/precedents` הקיים).
- **קבלה:** עו"ד מוצא פסק/סעיף, קורא מילולית, ומצטט לתיק — בלי לצאת מהמערכת.

## Phase F-C — מילון מונחים מרכזי + חילוץ רכיבים משותפים *(שבועות 2–3, refactor)*
**הבסיס שדולג:** הלוגיקה קיימת אך מפוזרת inline. לאחד למקור-אמת אחד.
- [ ] **`apps/dashboard/src/lib/legal-terms.ts`** — לרכז את כל מיפויי-התוויות הקיימים
      (STATUS_LABELS / PROC_LABELS / STATE_LABELS / risk BANDS / citation STATUS) למקום אחד; כל מסך צורך ממנו.
- [ ] **לחלץ רכיבים משותפים** מהקוד ה-inline הקיים ל-`components/common/`:
      `<DeadlineChip>`, `<SourceLink>`, `<ConfidenceBadge>` (כבר קיים inline ב-PrecedentsPage → לייצא),
      `<EmptyState>`, `<HebrewDate>`, `<CaseNumber>`.
- [ ] **`<AsyncBoundary>`** חדש (טעינה/שגיאה/ריק אחיד) — לעטוף בו מסכים בהדרגה.
- **קבלה:** אפס מיפוי-תוויות מוכפל; ≥4 מסכים צורכים את הרכיבים המשותפים והמילון.

## Phase F-D — דף בית "היום שלי" (מבוסס-מטלה) *(שבועות 3–4)*
החלפת `DashboardPage` ה-KPI-טכני במסך ממוקד-פעולה.
- [ ] בלוקים לחיצים: מועדים דחופים היום/השבוע · משימות · דיונים קרובים · מסמכים שנקלטו וממתינים לאישור · התראות.
- [ ] כל פריט מוביל ישירות לפעולה/ליעד; שימוש ב-`<DeadlineChip>`/`<EmptyState>` מ-F-C.
- [ ] לשמר תצוגת-KPI כטאב/אזור משני ("סקירת מערכת") למי שרוצה.
- **קבלה:** עו"ד נכנס בבוקר ורואה "מה דחוף היום" בלי לחפש.

## Phase F-E — הכרעת שולחן-עבודה (3-פאנלים) *(החלטת-עיצוב, שבוע 4)*
שולחן 3-הפאנלים (ציר\|מסמך\|תובנות) **לא קיים**. אך CaseDetail (טאבים) + Canvas + HearingPrep מכסים חלק.
- [ ] **החלטה דרושה:** לבנות `workbench` ייעודי, *או* להרחיב את `CanvasPage`/`CaseDetail` לתצוגת-split.
- [ ] אם בונים: layout split-pane (ציר-זמן | צופה-מסמך | תובנות) עם מצבי-מסך מותאמים.
- **קבלה:** עו"ד עובד על מסמך תוך ראיית ציר-הזמן והתובנות בו-זמנית — בלי קפיצות-ניווט.

## Phase F-F — אחידות נתיב-אישור-AI + מקור בכל מקום *(שבוע 5)*
חלקים קיימים (Show Source, ConfidenceBadge inline). להפוך לאחיד-מערכת.
- [ ] **זרימת "אשר / דחה / ערוך + מקור"** אחידה לכל פלט-AI (action-plan, תובנות, תבניות, Stens).
- [ ] כל תובנה עם רמת-ודאות + "מקור" + באנר "דרושה בדיקת עו"ד" היכן שמתחייב (מועדים/סיכון).
- [ ] להחליף כל טקסט-דמו מקודד שנותר (אם קיים) בפלט אמיתי.
- **קבלה:** אין פלט-AI בלי מקור+ודאות+נתיב-אישור.

## Phase F-G — ליטוש: נגישות, הדפסה, מצבי-מצב, ביצועים *(שבוע 6)*
- [ ] **הדפסה/ייצוא משפטי** — חיבור ה-docx הקיים ל-UI (כתבי-טענות, "תיק דיון", סיכומי-תיק).
- [ ] **נגישות (a11y)** — ניווט-מקלדת מלא, ARIA, ניגודיות, גודל-טקסט (קריטי לטקסט משפטי ארוך).
- [ ] **מצבי שגיאה/טעינה/ריק** אחידים דרך `<AsyncBoundary>` (F-C), בעברית, בלי jargon.
- [ ] **ביצועים** — virtualization לטבלאות גדולות, lazy-load למסכים כבדים, מדידת bundle.
- **קבלה:** lint+typecheck נקיים, ביקורת a11y עוברת, אין רגרסיית ביצועים.
